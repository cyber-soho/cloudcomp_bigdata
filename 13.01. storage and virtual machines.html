Virtual Disks in the Cloud

  • ▪️ Structure: Data split into fixed-size chunks (blocks), each with a numeric address.
  • ▪️ Access: Device-level (iSCSI/NVMe). Fastest access, highest cost.
  • ▪️ Attach Model: Typically single-attach (one volume per VM).
  • ▪️ Best For: Databases, OS boot volumes, low-latency applications.
🔑 Key Takeaway: Block storage is where your databases, Jupyter servers, and OS live. Master IOPS and throughput to avoid bottlenecks.

The Data Lake Backbone

  • ▪️ Structure: Flat namespace. Data stored as self-describing objects (data + metadata + unique ID).
  • ▪️ Access: API (HTTP/HTTPS). Immutable data (WORM pattern).
  • ▪️ Scalability: Near-infinite, cheapest at scale.
  • ▪️ Best For: Data lakes, machine learning training datasets, backups, archiving.
🔑 Key Takeaway: Object storage is your primary workhorse — it backs every data lake, ML pipeline, and analytics warehouse.

Network-Attached Storage (NAS)

  • ▪️ Structure: Traditional hierarchical directory tree.
  • ▪️ Access: Protocol (NFS/SMB). Shared access across networks.
  • ▪️ Attach Model: Multi-attach. Dozens to thousands of VMs can mount simultaneously.
  • ▪️ Best For: Shared datasets, distributed ML training, content management.
🔑 Key Takeaway: When multiple compute nodes need simultaneous access to the same dataset — file storage is your answer.

Storage Integration Workflow (Block)

  1. Attach — Create volume, attach to instance
  2. Format — e.g., sudo mkfs.ext4 /dev/xvdf
  3. Mount — e.g., sudo mount /dev/xvdf /mnt/data
  4. Persist — Add entry to /etc/fstab
⚠️ Common Mistake: Forgetting to terminate. A running GPU instance can cost ~$2,200/month. Stop pauses billing (mostly); Terminate ends it.
