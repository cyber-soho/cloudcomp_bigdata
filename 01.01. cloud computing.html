Introduction to Cloud Computing

This section establishes the foundational knowledge of what cloud computing is, why it is essential for modern data science, and the basic economic models governing it. You will explore the core NIST characteristics that define a true cloud and compare the costs of major providers.

💡 The Core Concept

Cloud computing = renting someone else's powerful computers over the internet, paying only for what you use, and scaling up or down whenever you want.

The Electricity Analogy: Nobody builds their own power plant when they open a bakery. They plug in and pay for what they consume. Cloud is the same idea — for compute, storage, and software.

The 5 Essential Characteristics (NIST)

Click each characteristic to reveal its definition. A true cloud service must have ALL five.

1. On-Demand Self-Service

Spin up a 32-core server in Singapore with a few clicks, no tickets, no waiting.

2. Broad Network Access

Accessible from any device over standard protocols.

3. Resource Pooling

Multi-tenancy: provider serves many customers simultaneously; resources dynamically reassigned.

4. Rapid Elasticity

Scale out (add more) or scale in (reduce) extremely quickly, often automatically.

5. Measured Service

Pay-as-you-go billing. Exact usage tracking. No more €50,000 servers running at 3% capacity.

Service Models: The Building Blocks

Infrastructure as a Service

What you get: Raw compute, storage, networking.

You manage: OS, middleware, runtime, application. Maximum control.

DS Use Case: Rent GPU server, install CUDA + Python + PyTorch yourself. (e.g., AWS EC2, Azure VMs)

Provider Management Level: Low

Platform as a Service

What you get: Infrastructure managed for you; you just bring your code.

DS Use Case: Write Spark job, Databricks handles the underlying clusters and OS. (e.g., Databricks, Google App Engine)

Provider Management Level: Medium

Software as a Service

What you get: Fully managed software via browser or API.

DS Use Case: Log ML experiments to Weights & Biases, absolutely zero server management required. (e.g., Snowflake, W&B)

Provider Management Level: High (Complete)

Cloud Provider Cost Comparison

Approximate monthly cost for 1 TB Storage + 3× Full Retrieval. Notice how egress (outbound data transfer) often costs more than the storage itself. Design your architecture to minimise data movement.

Storage Cost = price per GB × total GB
Egress Cost = (transfer GB − free tier) × egress rate

Regions & Availability Zones

  • Region: A distinct geographic area (e.g., eu-west-1 Ireland).
  • Availability Zone (AZ): One or more data centres within a region, with independent power/cooling.

Why geography matters for Data Scientists:

  1. Data Sovereignty: GDPR requires EU citizen data stays in EU regions.
  2. Latency: Deploying in eu-west-1 adds 200ms+ delay for Asian users.
  3. Disaster Recovery: Distributing across AZs survives single data centre failure.

Storage Types

Object Storage (e.g., AWS S3)

Infinitely scalable, cheap, flat structure. Best for: training datasets, model checkpoints.

Block Storage (e.g., AWS EBS)

Like a traditional hard drive attached to a VM. Faster, more expensive.

File Storage (e.g., AWS EFS)

Shared network file systems. Multiple compute instances access simultaneously.

Virtualization & Containerization

Solving the notorious "It works on my machine" problem. This section compares legacy Virtual Machines with modern Containerization strategies (Docker, Kubernetes), which are critical for ensuring machine learning models perform identically in development and production.

Virtual Machine (VM)

Abstracts the Hardware

A software-emulated computer running on top of physical hardware via a Hypervisor. Includes a full guest Operating System.

  • Size: Gigabytes (GBs)
  • Boot Time: Minutes
  • Isolation: Strong (Full OS)
  • Analogy: Complete Apartment

Container

Abstracts the Operating System

Packages application and dependencies into a lightweight unit that shares the host OS kernel. No guest OS inside.

  • Size: Megabytes (MBs)
  • Boot Time: Milliseconds
  • Isolation: Good (Namespaces)
  • Analogy: Hotel Room

Resource Efficiency Comparison

Containers are 10–100× smaller than VMs. You can run hundreds of containers on hardware that might support only 20 VMs.

Containerising an ML Pipeline: The Pattern

FROM python:3.11-slim # Base image
COPY requirements.txt . # Enable layer caching
RUN pip install -r requirements.txt # Install dependencies
ENV PYTHONUNBUFFERED=1 # Real-time logs
CMD ["python", "train_model.py"] # Default command

Docker makes containers mainstream. Same image runs identically on laptop, CI/CD, and cloud.

Kubernetes (K8s) orchestrates them at scale. Manages hundreds of containers (Pods), handles auto-scaling, and rolling deployments.
