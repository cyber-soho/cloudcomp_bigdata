Chapter 2 — Day 10

Basics of Cloud Computing Using AWS

A deep dive into AWS (Amazon Web Services), the market leader commanding ~31–33% of global IaaS (Infrastructure as a Service) revenue according to Gartner and Synergy Research Group.

Origin & Scale

Launched by Jeff Bezos in 2006, AWS grew from Amazon's internal infrastructure needs into 200+ services. Its pay-as-you-go model transformed capital expenditure (CapEx) into operational expenditure (OpEx). A GPU-equipped EC2 (Elastic Compute Cloud) instance costs ~$0.526/hour versus ~$4,000 upfront for equivalent hardware.

AWS Global Infrastructure

Regions (geographic clusters of data centres), Availability Zones (AZs — isolated facilities within a region with independent power/cooling), and Edge Locations (for CDN — Content Delivery Network caching). The Shared Responsibility Model divides security duties: AWS secures the infrastructure; the customer secures data, access, and application-layer configurations. IAM (Identity and Access Management) is the service for managing users, roles, and permissions.

Pricing Models

On-Demand (pay-as-you-go), Reserved Instances (RIs — commit 1–3 years for up to 72% savings), Spot Instances (bid on unused capacity at steep discounts, but instances can be reclaimed), and the Free Tier (12 months of limited services plus always-free tiers).

Chapter 3 — Day 11

Basics of Cloud Computing Using MS Azure

Microsoft Azure, the second-largest cloud, launched in 2010 and rebranded from "Windows Azure" in 2014. It operates 60+ regions — more geographic coverage than any competitor — and offers 600+ services.

Why Azure Matters for Data Science

Three reasons stand out: seamless integration with the Microsoft ecosystem (Microsoft 365, Power BI, SQL Server); Azure Machine Learning provides managed Jupyter notebooks, auto-scaling GPU clusters, model registry, and REST (Representational State Transfer) API deployment; the exclusive partnership with OpenAI means Azure is the only enterprise cloud host for GPT (Generative Pre-trained Transformer) models.

Shared Responsibility Model

Regardless of service model (IaaS, PaaS — Platform as a Service, SaaS — Software as a Service), your data and access controls are always your responsibility. Physical infrastructure responsibility shifts to Microsoft as you move up the service model stack.

Infrastructure Concepts

Regions → Availability Zones → Resource Groups (logical containers for managing related resources). Azure Resource Manager (ARM) is the unified management layer for all requests. Access is via the Azure Portal (web GUI — Graphical User Interface), Azure CLI (Command Line Interface), PowerShell, ARM/Bicep templates (Infrastructure as Code), and SDKs (Software Development Kits).

Storage Tiers

Hot (frequently accessed), Cool (infrequently accessed — lower storage cost, higher access cost), Cold (rarely accessed), Archive (cheapest storage, retrieval takes hours). Blob Storage handles unstructured data; Data Lake Storage Gen2 adds hierarchical file system for big data analytics; Cosmos DB provides globally distributed NoSQL at single-digit-millisecond latency.

Certifications

AZ-900 (Azure Fundamentals), DP-100 (Data Scientist Associate), AI-102 (AI Engineer Associate). Cloud-certified professionals command significantly higher starting salaries. The Azure Pricing Calculator is essential for cost estimation.

Chapter 4 — Day 12

Basics of Cloud Computing Using GCP

GCP (Google Cloud Platform) is the third-largest cloud by revenue but leads in data analytics, ML, and container orchestration. Google's internal infrastructure (Jupiter networking, Borg/Kubernetes scheduler, Colossus file system) is the same hardware you rent when using GCP.

Timeline Highlights

2008: App Engine launches. 2011: Cloud Storage and BigQuery (beta). 2014: Google open-sources Kubernetes (K8s). 2017: Cloud Spanner (globally-distributed relational DB). 2023: Vertex AI Gemini integration.

Infrastructure

Zones (single data-centre locations with independent power/cooling/networking), Regions (geographic clusters of 3+ zones, <5 ms inter-zone latency), Multi-Regions/Dual-Regions (for maximum availability). As of 2024: 38+ Regions, 115+ Zones, 200+ countries. Google's private backbone carries ~40% of global internet traffic, providing premium-tier networking.

Flagship Service: BigQuery

A serverless, fully managed data warehouse scanning petabytes in seconds. Pricing: $6.25 per TB (Terabyte) scanned, with 1 TB free monthly. No infrastructure to manage, no indexes to build, no clusters to resize. The analytics ecosystem layers Dataflow (batch/stream processing), Dataproc (managed Spark/Hadoop), Pub/Sub (real-time messaging), and Looker (BI — Business Intelligence visualisation).

ML & AI

Vertex AI (unified ML platform), TensorFlow (world's most popular open-source ML framework, invented at Google), TPUs (Tensor Processing Units — purpose-built AI accelerators, often more cost-effective than NVIDIA GPUs for large model training). GCP is the birthplace of the Transformer architecture (the "T" in GPT).

Data Residency

GDPR may require EU citizen PII (Personally Identifiable Information) to stay within EU territory. Egress costs apply for inter-region data transfer (~$80 per TB from us-central1 to europe-west1). A smart architect co-locates data and compute in the same region.

Chapter 5 — Day 13

Storage and Virtual Machines

Three Storage Paradigms

ParadigmHow It WorksCloud ServicesData Science Use Case
Block StorageFixed-size blocks, numeric addresses, no metadata. Fastest, most expensive, typically single-attach.Amazon EBS (Elastic Block Store), Azure Managed Disks, Google Persistent DiskOS boot volumes, databases, Jupyter servers
Object StorageData + rich metadata + unique ID in a flat namespace. Cheapest at scale, higher latency, immutable (re-upload to edit).Amazon S3, Azure Blob Storage, Google Cloud Storage (GCS)Data lakes, ML training datasets, log archives
File StorageHierarchical files/folders via NFS (Network File System) or SMB (Server Message Block). Multi-attach, familiar.Amazon EFS (Elastic File System), Azure Files, Google FilestoreShared datasets for distributed ML training

Block Storage Deep Dive

Performance measured in IOPS (Input/Output Operations Per Second) and throughput (MB/s). AWS EBS gp3 baseline: 3,000 IOPS included; additional IOPS at $0.005/IOPS-month. Pricing example: 200 GB gp3 at 6,000 IOPS = $31/month.

Object Storage Concepts

Objects in a flat namespace (forward slashes are part of the key string, not real directories). WORM (Write Once Read Many) pattern. HeadObject API retrieves metadata without downloading data. Storage tiers: Standard → Infrequent Access (IA) → Glacier/Archive.

Virtual Machines & Hypervisors

A VM (Virtual Machine) is a software-defined computer with emulated CPU, RAM, storage, and NIC (Network Interface Card). Type 1 (bare-metal) hypervisors (VMware ESXi, Microsoft Hyper-V, KVM — Kernel-based Virtual Machine) run directly on hardware — used by all major clouds. Type 2 (hosted) hypervisors (Oracle VirtualBox, VMware Workstation) run on a host OS.

VM Lifecycle

Pending → Running → Stopped → Terminated. Resizing requires stop-change-start. Always separate OS volume from data volume. AMI (Amazon Machine Image) is the AWS template for creating instances.

Snapshots & Disaster Recovery (DR)

Incremental snapshots store only changed blocks. Example: 500 GB volume with 10 GB/day changes — full daily backups = 15,000 GB/month ($750); incremental snapshots = 790 GB/month ($39.50) — a 95% reduction. RPO (Recovery Point Objective) = max acceptable data loss. RTO (Recovery Time Objective) = max acceptable downtime.

Chapter 6 — Day 14

Big Data, Distributed Systems, and Apache Spark (with Hadoop)

The Five V's of Big Data

Framework originally proposed by Doug Laney at Gartner: Volume (PB/EB scale), Velocity (real-time streaming), Variety (structured/semi-structured/unstructured), Veracity (data quality and trustworthiness), Value (business insights derived after processing).

Scaling Paradigms

Vertical scaling ("scale up"): bigger, more expensive single machine — hard hardware ceilings, SPOF (Single Point of Failure). Horizontal scaling ("scale out"): many commodity machines networked together — the foundation of Hadoop, Spark, and all cloud computing.

Distributed Systems & CAP Theorem

Eric Brewer (UC Berkeley) formalised in 2000: any distributed data store can guarantee at most two of three properties simultaneously — Consistency (every read returns latest write), Availability (every request gets a response), Partition Tolerance (survives network splits). In practice, choose CP (e.g., HBase, MongoDB) or AP (e.g., Cassandra, DynamoDB). Key concepts: Replication, Sharding (Partitioning), Consensus Algorithms (Paxos, Raft), Fault Tolerance.

Apache Hadoop

Created by Doug Cutting at Yahoo! based on Google's GFS (2003) and MapReduce (2004) papers. Donated to ASF (Apache Software Foundation). Core components: HDFS (Hadoop Distributed File System) — splits files into 128 MB blocks, replicates 3×; YARN (Yet Another Resource Negotiator) — cluster resource manager; MapReduce — two-phase processing (Map → Shuffle & Sort → Reduce). Ecosystem: Hive (SQL-on-Hadoop), HBase (NoSQL), Kafka (streaming), ZooKeeper (coordination), Sqoop (DB transfer), Flume (log ingestion).

HDFS Architecture

NameNode (master): maintains file system namespace and block-to-DataNode metadata map. DataNodes (workers): store actual data blocks, send heartbeats. Secondary NameNode: merges edit log with filesystem image (not a backup).

Apache Spark

Created by Matei Zaharia at UC Berkeley's AMPLab (2009), donated to ASF in 2013. Key innovation: RDD (Resilient Distributed Dataset) — immutable, distributed collections in memory, up to 100× faster than disk-based MapReduce. Components: Spark Core (scheduling, fault recovery), Spark SQL (DataFrames), Spark Streaming, MLlib (Machine Learning Library), GraphX. Lazy evaluation: Transformations (map, filter, groupByKey) define what to compute; Actions (collect, count, reduce) trigger execution.

Hadoop vs. Spark

CriterionHadoop (MapReduce)Apache Spark

The modern reality: many organisations run Spark on top of Hadoop's YARN and HDFS — Hadoop provides storage/resource management, Spark provides fast processing.

Industry Pipelines

Netflix: Spark + Kafka + Cassandra (500 billion events/day). Uber: Spark + Hadoop + Hive (surge pricing, fraud detection). Spotify: Spark + BigQuery (Discover Weekly playlists). NASA (National Aeronautics and Space Administration): Hadoop + Spark (satellite imagery). CERN (European Organisation for Nuclear Research): Spark + Hadoop (particle collision data from the LHC — Large Hadron Collider).

Case 1 — Ch.1 (Day 09) & Ch.4 (Day 12)

European Fintech Choosing a Cloud Provider

A Berlin-based fintech startup processes payment data from French and German customers. Their legal team confirms all transaction data must remain within the EU under GDPR. They need a cloud provider for a fraud-detection ML pipeline. Budget is tight, the team has Python expertise, and they need serverless analytics. Which cloud provider and region should they choose?

Recommended Answer

GCP with BigQuery, deployed in europe-west1 (Belgium) or europe-west3 (Frankfurt). Reasoning: GCP's BigQuery offers serverless, pay-per-TB analytics ideal for budget-conscious startups. Both named regions are within the EU, satisfying GDPR data-residency requirements. Vertex AI integrates natively for the ML fraud-detection model. Alternative: Azure in West Europe (Netherlands) would also comply with GDPR and offers Azure ML, but BigQuery's per-TB pricing model is more cost-effective for analytics-heavy workloads. AWS eu-central-1 (Frankfurt) is another valid choice but less compelling for analytics-first workflows.

Case 2 — Ch.2 (Day 10)

Graduate Research Lab Cost Estimation

A data science research lab needs to train a customer churn model on AWS. They require a GPU-equipped EC2 p3.2xlarge instance ($3.06/hr for 160 hours/month), a t3.micro API endpoint (730 hrs/month), 500 GB S3 Standard storage, 2 TB S3 Glacier archive, and 5 million Lambda invocations. What is the estimated monthly cost, and what is the dominant expense category?

Recommended Answer

EC2 p3.2xlarge: $3.06 × 160 = $489.60. EC2 t3.micro: $0.0104 × 730 = $7.59. S3 Standard: $0.023 × 500 = $11.50. S3 Glacier: $0.004 × 2,000 = $8.00. Lambda: $0.0000002 × 5,000,000 = $1.00. Total: ~$517.69/month. Compute (EC2) dominates at 96% of the total bill. The key insight: always monitor GPU instance runtime because leaving a GPU instance running overnight can double costs. Reserved Instances (RIs) can save up to 72% on EC2.

Case 5 — Ch.5 (Day 13)

Choosing the Right Storage Paradigm

A data engineering team has three distinct workloads: (A) a PostgreSQL database serving real-time queries for a web application; (B) a 50 TB data lake of historical sensor data for ML training; (C) a shared training image dataset that three GPU instances need to read simultaneously. Which cloud storage paradigm should be used for each workload?

Recommended Answer

(A) Block Storage (Amazon EBS / Azure Managed Disks / Google Persistent Disk) — PostgreSQL needs fast, low-latency, single-attach storage with high IOPS. (B) Object Storage (Amazon S3 / Azure Blob / GCS) — 50 TB of historical data fits the WORM pattern; object storage is the cheapest at scale with rich metadata for data governance. (C) File Storage (Amazon EFS / Azure Files / Google Filestore) — the multi-attach requirement rules out block; the simultaneous-read pattern from multiple GPU instances requires NFS/SMB shared access. The key trade-off is cost vs. access pattern vs. multi-tenancy.

Case 6 — Ch.5 (Day 13)

Snapshot Strategy for a Production Database

A 500 GB PostgreSQL volume changes about 10 GB per day. The operations team must design a backup strategy with an RPO (Recovery Point Objective) of 1 hour and an RTO (Recovery Time Objective) of 15 minutes. Compare full daily backups versus incremental snapshots over a 30-day retention window.

Recommended Answer

Incremental snapshots every hour using AWS Data Lifecycle Manager (or Azure Backup / GCP Scheduled Snapshots). Full daily: 500 × 30 = 15,000 GB at $0.05/GB = $750/month. Incremental: 500 + (29 × 10) = 790 GB = $39.50/month — a 95% cost reduction. For the 1-hour RPO, hourly snapshots ensure at most 1 hour of data loss. For the 15-minute RTO, create a new volume from the latest snapshot and attach it to a standby VM — typically achievable in under 15 minutes. Cross-region snapshot replication provides disaster recovery for regional outages.

Case 7 — Ch.6 (Day 14)

SmartCampus University — Five V's Analysis

A university with 30,000 students collects 2 TB of lecture recordings, 500 GB of assignment submissions, and 50 GB of admin records per semester. During exam week, the LMS (Learning Management System) logs 10,000 click-events per minute. Analyse this scenario through the Five V's framework.

Recommended Answer

Volume: ~2.55 TB per semester across multiple formats. Velocity: 10,000 click-events/minute during exam week — requires real-time or near-real-time ingestion (Kafka / Pub/Sub). Variety: Structured (grade spreadsheets), semi-structured (JSON API logs), unstructured (discussion forum text, video recordings). Veracity: Typos in student names, double-counted attendance scans, missing fields — cleaning required before analysis. Value: Correlating lecture topics with poor exam scores enables curriculum redesign, improving pass rates by 12%. A suitable pipeline: ingest via Kafka → store in HDFS/S3 → process with Spark → analyse with Spark SQL/MLlib → visualise in Looker/Tableau.

Case 8 — Ch.6 (Day 14)

HDFS Block Distribution for a Large File

A 512 MB file named "sales_data.csv" is stored on an HDFS cluster with 5 DataNodes, a block size of 128 MB, and a replication factor of 3. How many blocks are created, how many total block replicas exist across the cluster, and what happens if one DataNode fails?

Recommended Answer

512 MB / 128 MB = 4 blocks. Replication factor 3 means each block is stored on 3 different DataNodes, so total replicas = 4 × 3 = 12 block copies distributed across 5 DataNodes. If one DataNode fails, each block still has at least 2 surviving copies on the remaining nodes — the data is safe. The NameNode detects the failure via missing heartbeats and instructs surviving DataNodes to re-replicate under-replicated blocks to restore the replication factor to 3. This is HDFS's core fault-tolerance mechanism.

Case 9 — Ch.6 (Day 14)

Spark vs. Hadoop for ML Training

Your team needs to train an ML model that requires 100 iterations over the same dataset. Compare the performance implications of using Hadoop MapReduce versus Apache Spark for this workload. Which should you choose and why?

Recommended Answer

Apache Spark is the clear winner for iterative ML workloads. Hadoop MapReduce writes intermediate results to disk after every Map and Reduce step. For 100 iterations, this means 200+ disk write/read cycles — devastating for performance. Spark keeps data in memory via RDDs (Resilient Distributed Datasets), so subsequent iterations access RAM instead of disk — up to 100× faster. Spark's built-in MLlib provides classification, regression, and clustering algorithms, while Hadoop requires external tools like Apache Mahout. Hadoop remains relevant for massive one-time batch ETL on cold archival data where disk cost is lower than memory cost. The modern best practice: run Spark on YARN over HDFS — Hadoop provides storage, Spark provides fast iterative processing.

Case 10 — Ch.6 (Day 14) & Ch.2 (Day 10)

End-to-End Big Data Pipeline Design

An e-commerce company needs a pipeline that ingests real-time clickstream data, stores it for analytics, processes it to detect purchasing patterns, trains an ML recommendation model, and serves predictions to users. Design this pipeline using cloud-native services, specifying at least one tool for each stage.

Recommended Answer

Ingestion: Apache Kafka or Amazon Kinesis for real-time event streaming. Storage: Amazon S3 or Google Cloud Storage (GCS) for raw data in Parquet format (object storage for data lake). Processing: Apache Spark (via AWS EMR, Azure HDInsight, or GCP Dataproc) for batch and stream processing. Analysis & ML: Spark MLlib or Amazon SageMaker for recommendation model training. Serving: A REST API endpoint deployed on AWS Lambda or Azure Functions for low-latency prediction serving. Visualisation: Looker, Tableau, or Power BI for business dashboards. The pipeline follows the canonical pattern: Sources → Ingest → Store → Process → Analyse → Visualise. Cost optimisation: use S3 Intelligent-Tiering for automatic tier management, and Spot Instances for Spark cluster workers to reduce compute costs.
