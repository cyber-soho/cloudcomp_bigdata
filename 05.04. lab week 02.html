The NIST ↗ identifies five essential characteristics of cloud computing. These are the baseline reference you will use throughout your career.

01
On-Demand Self-Service
Provision resources without human interaction from the provider
02
Broad Network Access
Resources accessible from any device, anywhere via the internet
03
Resource Pooling
Multiple clients share underlying infrastructure (multi-tenancy)
04
Rapid Elasticity
Capacity scales automatically up or down with demand
05
Measured Service
Usage is metered; you pay for exactly what you consume

Public Cloud
Third-party provider. Any paying customer. Scalable web apps & storage.
Private Cloud
Single organization. Internal teams only. Sensitive regulated data.
Hybrid Cloud
Public + private mix. Burst workloads + compliance requirements.
Multi-Cloud
Multiple public providers. Best-of-breed services. Avoid vendor lock-in.
Community Cloud
Shared by sector group. Government, healthcare consortiums.

Step 2.3 — Virtualization: The Engine Behind the Cloud

Hypervisor+

A hypervisor is a software layer that creates and manages VMs on physical hardware. Type 1 hypervisors run directly on bare metal (e.g., KVM, Xen, VMware ESXi). Type 2 run atop a host OS (e.g., VirtualBox, VMware Workstation).

Virtual Machine (VM)+

A VM (Virtual Machine) is a software emulation of a complete computer, running its own OS. Multiple VMs can coexist on a single physical server, each fully isolated.

Container+

A container is lightweight virtualization that shares the host OS kernel but isolates application processes. Containers start in seconds, use far less memory than VMs, and are highly portable via images.

Orchestration+

Orchestration is the automated management of many containers or VMs across a cluster — handling scheduling, scaling, networking, and self-healing. Kubernetes (K8s — kubernetes.io ↗) is the dominant open-source standard.

On Containers — Open Tools
  • Docker ↗ — most widely known containerization platform
  • Podman ↗ — daemonless Docker alternative (open-source)
  • k3s ↗ — lightweight Kubernetes distribution

⚠️ Any equivalent, newer, or self-created tool is equally valid for this lab.

Step 3.1 — What Is Cloud Governance and Why Does It Matter?

Cloud governance is the set of policies, procedures, roles, and technical controls that an organization establishes to manage its cloud resources in alignment with its strategic objectives, security requirements, regulatory obligations, and financial constraints.

Without governance, cloud environments tend toward:

  • Shadow IT: Employees provisioning unauthorized resources without oversight.
  • Cost overruns: Uncontrolled resource sprawl and forgotten instances.
  • Security gaps: Misconfigured storage, excessive permissions, unpatched systems.
  • Compliance violations: Failing to meet legal or sector-specific data regulations.
Reference Frameworks (Starting Points — Not Mandatory)
  • Cloud Security Alliance (CSA) ↗ — Cloud Controls Matrix (CCM)
  • European Union Agency for Cybersecurity (ENISA) ↗ — Cloud governance guidelines
  • ISO/IEC 27001 ↗ — Information security management standard
  • GDPR (General Data Protection Regulation) ↗ — EU data protection law

⚠️ Students are encouraged to find equivalent or complementary frameworks, local regulatory bodies, or sector-specific standards.

Step 3.2 — The Four Pillars of Cloud Governance

💰
Financial Governance (FinOps)
  • Tag all resources with cost center, project, owner
  • Set budgets and automated alerts
  • Right-size instances regularly
  • Review idle/unused resources

→ FinOps Foundation ↗

🔒
Security & Compliance
  • Apply PoLP — minimum permissions always
  • Use IAM policies rigorously
  • Encrypt data at rest and in transit
  • Audit access logs; enable threat detection
⚙️
Operational Governance
  • Use IaC for reproducible deployments
  • Define naming conventions & tagging standards
  • Establish change management procedures
  • Maintain architecture documentation
⚠️
Risk & Compliance
  • Vendor lock-in risk
  • Data residency & jurisdiction risk
  • Provider availability risk
  • Supply chain vulnerability risk
