Compute Paradigms

Pick the right tool for your workload.

Compute Engine (IaaS)

Full VM root access. Best for custom CUDA kernels or PySpark clusters.

App Engine (PaaS)

Push code, Google scales. Best for lightweight ML API endpoints.

Cloud Run (CaaS)

Serverless containers. Scales to zero. Best for bursty ML traffic.

GKE (Kubernetes)

Managed cluster. Best for complex MLOps pipelines (Kubeflow).

BigQuery: The Analytics Engine

BigQuery is Google's serverless data warehouse and a cornerstone for data scientists. This module unpacks why BigQuery is an OLAP query engine, not a traditional relational database. You will explore the profound performance and cost differences between row-based and columnar storage, and learn how to run Machine Learning models directly within SQL.

BigQuery is NOT a Database

It is a serverless query engine bolted onto a distributed file system, optimized for OLAP (Analytical) workloads, not OLTP (Transactional).

Traditional (Spark/Hadoop)
  • ❌ Provision cluster (10-15m)
  • ❌ Configure memory
  • ❌ Monitor node failures
BigQuery (Serverless)
  • ✅ Execute SQL instantly
  • ✅ Auto-allocates compute
  • ✅ Transparent infra

BigQuery ML (BQML)

Train models directly in SQL without moving data.

CREATE OR REPLACE MODEL `project.dataset.my_model`
OPTIONS(model_type='random_forest_classifier') AS
SELECT feature_1, feature_2, target_label
FROM `project.dataset.training_data`;

Supports Logistic Regression, K-Means, Random Forest, XGBoost, and ARIMA. The model lives inside BigQuery, accessible via ML.PREDICT().

Architecture: Row vs Columnar

BigQuery charges $5 per TB scanned. Columnar storage is ~7x cheaper/faster for analytics because it only reads the columns requested.

Row-Based (PostgreSQL/MySQL)

ID:1
Amt:$50
Cat:A
Date:1/1
ID:2
Amt:$20
Cat:B
Date:1/2

Querying `Amount` loads the entire row into memory. High I/O.

Columnar (BigQuery)

ID:1, 2
Amt:$50, $20
Cat:A, B
Date:1/1, 1/2

Querying `Amount` only reads the Amount block. Minimal I/O. Never use SELECT *.

IAM: The Three Pillars

Always apply the Principle of Least Privilege. Use Workload Identity Federation instead of downloading JSON keys.

1

Principal (Who)

Who is requesting access? e.g., Service Account, ML Engineer

2

Role (What)

What can they do? e.g., roles/bigquery.dataEditor

3

Resource (On What)

What are they accessing? e.g., Specific BQ Dataset
