Welcome to the foundation of modern cloud architecture. Service Oriented Architecture (SOA) is the paradigm that allows massive tech companies to have thousands of engineers working in parallel. Protocols & Integrations

How do services actually talk to each other? This section breaks down the historical heavyweight SOAP protocol versus the modern REST architectural style, and introduces the Enterprise Service Bus (ESB) which manages complex inter-service messaging.

SOAP vs REST

REST has become the default for data scientists and modern web apps due to its lightweight nature. SOAP remains in legacy enterprise (banking, healthcare) where formal contracts are non-negotiable.

For the same payload, SOAP is typically 8.2x larger than REST due to XML envelope verbosity.

REST (Modern Default)

  • ✓ Uses standard HTTP verbs (GET, POST, PUT, DELETE)
  • ✓ Lightweight JSON payloads
  • ✓ Stateless by design
  • ✓ Best for: ML model endpoints, web APIs, cloud data pipelines

SOAP (Legacy/Enterprise)

  • ✓ Strict XML `` structure
  • ✓ Built-in WS-Security & Transaction support
  • ✓ Mandatory WSDL contracts
  • ✓ Best for: Bank transactions, HL7 FHIR healthcare integrations

The Enterprise Service Bus (ESB)

Without an ESB, connections grow exponentially as point-to-point hell. The ESB acts as the airport baggage conveyor belt, managing routing, transformation, and security centrally.

SOA

The Outer Layer (Enterprise)

  • • Connects disparate enterprise systems (e.g., SAP to Salesforce)
  • • Coarse-grained services
  • • Often relies on ESB for communication
  • • Shared databases are common

Microservices

The Inner Layer (Application)

  • • Decomposes a single application into small services (e.g., Netflix)
  • • Fine-grained, independently deployable
  • • Direct, decentralised communication
  • • Each service strictly owns its own database
Verdict: They are not enemies. Most real systems are a hybrid (Microservices inside domains, SOA across domains).

SOA Security Fundamentals

Security is not a feature added at the end; it is an architectural property designed from the start. This section covers the three pillars of service security and the non-negotiable checklist for deploying ML APIs.

1. Authentication

"Who are you?"

  • API Keys: Low security (internal)
  • JWT: High security, stateless
  • OAuth 2.0: Delegated auth

2. Authorization

"What can you do?"

  • RBAC: Role-Based access
  • ABAC: Attribute-Based access
  • Principle: Least Privilege

3. Transport

"Can they eavesdrop?"

  • HTTPS/TLS: Mandatory encryption
  • mTLS: Mutual TLS (both sides verified)
  • Service Mesh: Enforces mTLS by default

✅ The Non-Negotiables for ML APIs

✓

HTTPS enforced — never HTTP in production

✓

Input validated — never trust incoming data (Injection risks)

✓

Authentication required — minimum API key, preferably JWT

✓

Logging active — log requests with user ID and timestamp

✓

Authorization checked — validate scope per endpoint

✓

Dependencies audited — check OWASP Top 10 before launch

✓

Rate limiting enabled — protect against abuse and cloud cost overruns
