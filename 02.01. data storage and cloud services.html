Data Storage

SECTION 1.1

What Is Data Storage? — The Foundation of Everything

Data storage refers to the recording and preservation of digital information using technology specifically designed for keeping that data accessible for future use. Every photograph on your phone, every transaction in your bank account, every heartbeat recorded by a smartwatch — all of this is data that must be stored somewhere.

At the hardware level, storage devices translate information into electromagnetic, optical, or electronic signals. At the software level, storage systems organize those signals into files, records, tables, and objects that applications can read and write efficiently.

Why Should a Data Scientist Care About Storage?

Speed
How fast you can query 10 billion rows
Cost

Real-Life Example — Your Smartphone as a Storage System

Understand data storage by examining a device you already own

Your smartphone contains between 64 GB (Gigabytes) and 1 TB (Terabyte) of storage. When you take a photograph, the camera sensor captures light, the processor compresses it into JPEG (Joint Photographic Experts Group) or HEIF (High Efficiency Image File Format), and the operating system writes that file to NAND (Negative-AND logic gate) flash memory. The file system — APFS (Apple File System) on iPhones or ext4 (Fourth Extended File System) on Android — keeps an index so you can find that photo later without searching every cell.

Python simple_storage_simulator.py
# Import modules for file system operations, JSON serialization, and timestamps
import os, json
from datetime import datetime

class SimpleStorageSystem:
    def __init__(self, storage_path="my_data_store"):
        # Create the storage directory (like provisioning a cloud bucket)
        self.storage_path = storage_path
        os.makedirs(self.storage_path, exist_ok=True)

    def store(self, filename, data):
        # Write data with metadata (timestamp, size) as JSON
        record = {"metadata": {"created": str(datetime.now())}, "data": data}
        with open(os.path.join(self.storage_path, filename), 'w') as f:
            json.dump(record, f, indent=4)

    def retrieve(self, filename):
        # Read data back from disk by filename
        with open(os.path.join(self.storage_path, filename), 'r') as f:
            return json.load(f)["data"]

# Demo: store, retrieve, and list files
storage = SimpleStorageSystem()
storage.store("student.json", {"name": "Alice", "grade": "A"})
print(storage.retrieve("student.json"))
  1. Imports: os for file system operations, json for data serialization, datetime for timestamps — standard Python libraries.
  2. Class Definition: SimpleStorageSystem bundles all storage logic, mirroring how cloud storage services expose an API (Application Programming Interface).
  3. store() — Writing Data: Wraps data with metadata and writes it as JSON. This illustrates the "Create" operation in CRUD (Create, Read, Update, Delete).
  4. retrieve() — Reading Data: Looks up a file by name, deserializes it, and returns the data. This is the "Read" operation.
  5. Demo Block: Creates a storage instance, stores a student record, and retrieves it — showing the complete lifecycle.
Key Takeaway Storage is not just an IT concern — it directly shapes what analytics are possible and affordable. The same write-index-retrieve principles apply from phones to petabyte data centers.
SECTION 1.2

The Three Fundamental Storage Architectures

ArchitectureAnalogyBest ForCloud Examples
Block StorageBlank hard drive split into chunksDatabases, VMs, OS boot volumesAWS EBS (Elastic Block Store), Azure Managed Disks
File StorageShared filing cabinetCollaboration, home directories, mediaAWS EFS (Elastic File System), Azure Files, NFS (Network File System)
Object StorageWarehouse of labeled boxesData lakes, backups, ML datasets, media archivesAWS S3, Azure Blob, GCS

Block Storage

Fixed-size blocks (512 B — 4 KB), unique addresses, lowest latency (~1 ms), highest IOPS (Input/Output Operations Per Second) — up to 64,000. Used by PostgreSQL, MySQL.

File Storage

Hierarchical directories and files. Access via NFS (Network File System) or SMB (Server Message Block). Multiple users read/write simultaneously. Most intuitive model.

Object Storage

Each object = data + metadata + unique ID. Flat namespace in buckets. Scales to petabytes. 99.999999999% durability (eleven nines). Backbone of the modern data lake.

Real-Life Example — Netflix and the Three Storage Types

How a streaming platform uses all three architectures simultaneously

Netflix stores petabytes of video in Object Storage (S3) for cheap, durable content delivery. Their ML recommendation engine uses Block Storage (EBS) for fast database I/O. Internal teams share configs via File Storage (EFS) for concurrent collaboration.

import random

# Performance profiles based on real cloud benchmarks
PROFILES = {
    "Block (EBS)":  {"latency_ms": 1.0,  "cost_gb": 0.10,  "iops": 64000},
    "File (EFS)":   {"latency_ms": 5.0,  "cost_gb": 0.30,  "iops": 35000},
    "Object (S3)":  {"latency_ms": 20.0, "cost_gb": 0.023, "iops": 5500},
}

# Simulate 5 reads per type with ±30% variance
for name, p in PROFILES.items():
    reads = [round(p["latency_ms"] * random.uniform(0.7, 1.3), 2) for _ in range(5)]
    cost_500gb = p["cost_gb"] * 500
    print(f"{name:15} | Latencies: {reads} ms | 500GB: ${cost_500gb:.2f}/mo")
Key Takeaway No single storage type does everything. Block = fastest (databases). Object = cheapest (data lakes). File = most collaborative (shared access). Choose by workload.

import os

# Map file extensions to data categories
EXT_MAP = {
    ".csv": "Structured", ".parquet": "Structured", ".sql": "Structured",
    ".json": "Semi-Structured", ".xml": "Semi-Structured", ".log": "Semi-Structured",

# Classify a list of hospital files
for f in ["patient_records.csv", "mri_scan.jpg", "lab_results.json"]:

    category = EXT_MAP.get(ext, "Unknown")

SECTION 1.4

Relational Databases and SQL — The Workhorse of Structured Data

Invented by Edgar F. Codd at IBM (International Business Machines Corporation) in 1970, the relational model organizes data into tables where rows = records and columns = attributes. SQL (Structured Query Language), standardized by ANSI (American National Standards Institute) and ISO (International Organization for Standardization), remains the most widely used data query language in the world.

Core Concepts

Primary Key: Unique identifier per row (e.g., StudentID)
Foreign Key: Links tables via referential relationships
JOINs: Reconstruct complex relationships from simple tables
Normalization: Reduce redundancy, improve integrity

ACID (Atomicity, Consistency, Isolation, Durability) Properties

A
Atomicity — All or nothing
C
Consistency — Valid state → valid state
I
Isolation — Concurrent txns don't interfere
D
Durability — Committed data survives crashes

Real-Life Example — University Registration System

The database behind your enrollment in this very course

Python + SQL university_registration_db.py
import sqlite3

conn = sqlite3.connect(":memory:")
cur = conn.cursor()
cur.execute("PRAGMA foreign_keys = ON")

# Create tables with primary and foreign keys
cur.execute("""CREATE TABLE students (
    student_id INTEGER PRIMARY KEY, first_name TEXT, last_name TEXT)""")
cur.execute("""CREATE TABLE courses (
    course_id INTEGER PRIMARY KEY, name TEXT, credits INTEGER)""")
cur.execute("""CREATE TABLE enrollments (
    student_id INTEGER, course_id INTEGER, grade TEXT,
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id) REFERENCES courses(course_id))""")

# Insert sample data
cur.execute("INSERT INTO students VALUES (1, 'Alice', 'Moreau')")
cur.execute("INSERT INTO courses VALUES (1, 'Cloud Computing', 3)")
cur.execute("INSERT INTO enrollments VALUES (1, 1, 'A')")

# JOIN query — the fundamental power of relational databases
cur.execute("""SELECT s.first_name, c.name, e.grade
    FROM enrollments e
    JOIN students s ON e.student_id = s.student_id
    JOIN courses c  ON e.course_id  = c.course_id""")
for row in cur.fetchall():
    print(f"{row[0]} | {row[1]} | Grade: {row[2]}")
Key Takeaway Relational databases + SQL have been the workhorse of structured data for 50+ years. Master JOINs and ACID — they appear in every data pipeline you will ever build.

The CAP Theorem (Eric Brewer, UC Berkeley, 2000)

Any distributed database can guarantee at most two of three:

C
Consistency
Every read returns latest write
A
Availability
Every request gets a response
P
Partition Tolerance
Works despite network splits

MongoDB → CP | Cassandra → AP | Traditional RDBMS → CA (not distributed)

Key Takeaway There is no "best" database. SQL excels at structured, transactional data. NoSQL excels at flexibility and scale. The CAP theorem explains why distributed systems always involve trade-offs.
SECTION 1.6

Data Lakes, Data Warehouses, and the Lakehouse

Data Warehouse

Cleaned, structured, optimized for SQL

  • ETL (Extract, Transform, Load) pipeline
  • Schema-on-write — structure enforced at load time
  • Fastest queries (pre-optimized)
  • Higher cost per GB
  • Best for: BI reporting, dashboards
  • Snowflake, Redshift, BigQuery

Data Lake

Raw, native format, all data types

  • ELT (Extract, Load, Transform) pattern
  • Schema-on-read — structure applied at query time
  • Variable query speed
  • Lower cost per GB
  • Best for: ML training, data exploration
  • AWS S3 + Glue, Azure Data Lake Storage

The Data Lakehouse — Best of Both Worlds

Combines lake storage costs with warehouse query performance. ACID guarantees on raw data. Pioneered by Databricks (Delta Lake), adopted by Apache Iceberg & Hudi. Rapidly becoming the new standard.

Key Takeaway Data lakes store everything cheaply. Data warehouses query fast. The lakehouse gives you both. Understand ETL vs. ELT — it defines your pipeline architecture.

Real-Life Example — A News Organization's Photo Archive

How Associated Press saves millions with lifecycle policies

Recent photos (30 days) → Standard. Last year → Infrequent Access. Past 5 years → Glacier. Pre-digital scans → Deep Archive. An S3 Lifecycle Policy automates all transitions — set rules once, save forever.

Python storage_tier_optimizer.py
# Calculate savings from storage tiering
TIERS = {
    "Standard":     0.023,
    "Infrequent":   0.0125,
    "Glacier":      0.004,
    "Deep Archive": 0.00099,
}

total_tb = 100
total_gb = total_tb * 1024
# Distribution: 10% hot, 20% warm, 30% cold, 40% archive
dist = {"Standard": 0.10, "Infrequent": 0.20, "Glacier": 0.30, "Deep Archive": 0.40}

all_standard = total_gb * TIERS["Standard"]
tiered = sum(total_gb * pct * TIERS[tier] for tier, pct in dist.items())
print(f"All Standard: ${all_standard:,.2f}/mo")
print(f"Tiered:       ${tiered:,.2f}/mo")
print(f"Savings:      ${all_standard - tiered:,.2f}/mo ({(1-tiered/all_standard)*100:.0f}%)")
Key Takeaway Storage tiering can reduce costs by 75%+. Always implement lifecycle policies — they are the single easiest cost optimization in cloud infrastructure.
SECTION 2.5

Serverless Computing — The Future of Cloud

Serverless computing: you write a function, define a trigger, and the cloud handles everything else. Pay only for actual compute time, measured in milliseconds. Scales from zero to millions and back to zero.

$0.0000002
Per 200ms execution (128 MB)
FREE
First 1M requests/month (Lambda)
0 → ∞ → 0
Auto-scales, zero idle cost

Ideal For

Event-driven image processing, API backends, data transformation triggers, IoT events, scheduled tasks (cron)

Not Ideal For

Long-running processes (15 min max), persistent connections (WebSockets), heavy ML training

Real-Life Example — Automatic Thumbnail Generation

Billions of executions daily across the internet

USER UPLOAD ──► S3 BUCKET ──► AWS LAMBDA ──► S3 OUTPUT
 photo.jpg      (triggers     (resizes to     photo_sm.jpg
  (5 MB)        Lambda)       150/300/600px)  photo_md.jpg
                                              photo_lg.jpg
                              Cost: ~$0.0000002 per execution
Python serverless_thumbnail_simulator.py
import time, json

def lambda_handler(event, context):
    # Standard Lambda entry point — receives event and context
    start = time.time()
    filename = event["filename"]
    sizes = [("sm", 150), ("md", 300), ("lg", 600)]

    for label, px in sizes:
        time.sleep(px / 10000)  # Simulate resize time
        print(f"  Generated {filename}_{label}.jpg ({px}x{px})")

    duration_ms = round((time.time() - start) * 1000, 2)
    cost = duration_ms * 0.0000000020833
    return {"duration_ms": duration_ms, "cost": f"${cost:.10f}"}

# Simulate invocation
result = lambda_handler({"filename": "vacation_photo.jpg"}, {})
print(json.dumps(result, indent=2))
Key Takeaway Serverless = pay only for what runs. Perfect for event-driven, bursty workloads. Not ideal for long-running processes (15 min max on Lambda).
SECTION 2.6

Cloud Security and Compliance — Shared Responsibility

Customer Secures (IN the Cloud)

Customer data & encryption config, IAM (Identity & Access Management) policies & MFA (Multi-Factor Authentication), application security, firewall rules, OS patching (IaaS)

"If you configured it wrong, it is YOUR fault."

Provider Secures (OF the Cloud)

Physical data centers, networking hardware, hypervisor / host OS, storage drive destruction, power/cooling, global network

"If it's in the data center, it is OUR responsibility."

Key Security Concepts

IAM: Who accesses what. Follow Principle of Least Privilege.
Encryption at Rest: AES-256 (Advanced Encryption Standard, 256-bit keys)
Encryption in Transit: TLS (Transport Layer Security)
VPC: Virtual Private Cloud — your isolated network
MFA: Multi-Factor Authentication — password + second factor
Compliance: SOC 2, HIPAA, GDPR, ISO 27001
🔓

Case Study — Capital One Data Breach (2019)

100M+ records exposed through misconfiguration, not infrastructure failure

A misconfigured WAF (Web Application Firewall) on Capital One's AWS infrastructure allowed an attacker to exploit SSRF (Server-Side Request Forgery), retrieve temporary IAM credentials, and download S3 data. AWS was NOT breached — the customer's configuration was.

Prevention: Least-privilege IAM, VPC endpoints, CloudTrail audit logging, regular security audits (AWS Security Hub, Prowler), encrypt everything.

Key Takeaway "Security OF the cloud" is the provider's job. "Security IN the cloud" is YOUR job. Most breaches come from misconfiguration, not infrastructure failure.
